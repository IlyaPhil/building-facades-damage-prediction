# Пайплайн построения Damage Index (до стадии GNN) — v3

Документ описывает полный пайплайн сбора данных и обучения моделей до момента интеграции с GNN.

**Ключевые принципы проектирования:**
- Устранение доменного сдвига через тайлизацию
- Единое 9-мерное признаковое пространство для всех зданий (без смешанных представлений)
- Попарное ранжирование как глобальный калибратор, ограниченное реалистичным временным бюджетом
- Гибридный вход в GNN (сырые ratios + ранговые scores) как резервный вариант
- Явно ограниченный scope антропогенного класса

# Фаза 1 — Сбор широких планов фасадов

Для каждого здания:

1. Открыть в режиме Яндекс Панорам.
2. Сделать скриншот **широкого плана фасада** (весь видимый фасад целиком).
3. Сохранить как: `images/wide/{building_id}.jpg`

## 1.1 Image-level метка (вспомогательная)

Немедленно при сборе присвоить зданию визуальную метку технического состояния:

| Метка | Описание | Визуальные признаки фасада |
|---|---|---|
| 0 | Повреждений не видно | Штукатурка цела, трещин нет, незначительные загрязнения |
| 1 | Незначительные повреждения | Локальные сколы, волосяные трещины, пятна на <30% поверхности |
| 2 | Значительные повреждения | Массовое отслоение штукатурки, трещины >1 мм, обнажение кладки на отдельных участках |
| 3 | Тяжёлые / аварийные | Обрушение штукатурки, глубокие трещины, обнажение кирпича на >60% поверхности |

Нормативная база шкалы: ГОСТ 31937-2024 (категории технического состояния), ВСН 53-86(р) (таблицы физического износа штукатурных фасадов).

Сохранить в: `image_level_labels.csv` (`building_id`, `condition_label`)

⚠ Эти метки — вспомогательный сигнал для валидации ранкера и слабой регуляризации. Они не являются структурным ground truth и не используются как обучающие таргеты.

# Фаза 2 — Создание датасета для сегментации

## 2.1 Стратегия отбора патчей

Целевой объём: **150–200 размеченных патчей** суммарно.

Распределение выборки:
- **125–170 патчей** из умеренно повреждённых зданий (КТС 0–1)
- **25–30 патчей** из тяжело повреждённых зданий (КТС 2–3)

Включение тяжёлых зданий критически важно для снижения доменного сдвига на инференсе. Качество разметки для тяжёлых случаев может быть грубее.

Для каждого выбранного здания:
- 2–4 кропа с одного здания
- Минимальное разрешение: 800×600 пикселей
- Стараться покрыть все три типа дефектов там где они присутствуют

Сохранить как: `images/crops/{building_id}_{n}.jpg`

## 2.2 Настройка проекта в CVAT

Тип задачи: **Semantic Segmentation** (режим Shape, не Track)

Классы:

| Класс | Описание |
|---|---|
| `background` | Неповреждённая поверхность фасада, окна, двери |
| `mechanical` | Трещины, сколы, отслоения штукатурки, структурные деформации |
| `biological` | Плесень, мох, высолы, пятна влажности |
| `anthropogenic` | Граффити, вандальные надписи, намеренные механические повреждения |
| `ignored` | Нечитаемые зоны (плотная растительность, сильная засветка и т.д.) |

⚠ **Scope класса `anthropogenic`**: исключить кондиционеры, вывески, рекламные щиты и инженерное оборудование — это модификации здания, не повреждения. Включать только граффити и вандализм.

## 2.3 Правила разметки

- Использовать **SAM2** для крупных связных дефектов (пятна, зоны отслоений).
- Использовать **ручной полигональный инструмент** для тонких линейных дефектов (трещины).
- Тяжёлые здания: размечать грубо, приоритет — охват, а не точность границ.
- Нечитаемые зоны помечать как `ignored` — они исключаются из расчёта loss.

Формат экспорта: PNG маски (предпочтительно) или COCO panoptic.

# Фаза 3 — Обучение модели сегментации

## 3.1 Архитектура

Рекомендуется: **SegFormer-B2** (хорошо работает из коробки на малых датасетах, эффективный инференс).

Альтернатива: DeepLabV3+ с backbone ResNet-50.

Предобучена на ImageNet. Выход: 4 класса (фон + 3 типа дефектов).

## 3.2 Функция потерь

```
CrossEntropyLoss(ignore_index=<индекс класса ignored>)
```

Опционально: взвешивание классов по частоте если дисбаланс существенный.

## 3.3 Аугментации

- Горизонтальный флип
- Color jitter
- Случайное кадрирование
- Вариация яркости и контраста

Имитируют вариативность условий съёмки панорам (освещение, угол, сезон).

## 3.4 Метрики

- mIoU по каждому классу
- mean mIoU

**Цель**: стабильная генерализация, не идеальная сегментация. Модель используется для относительного сравнения зданий, а не для точного пиксельного измерения.

# Фаза 4 — Извлечение признаков (все здания)

## 4.1 Процедура инференса

Для каждого здания:

**Если есть размеченные кропы:**
→ Прогнать сегментатор по всем доступным кропам.

**Если кропов нет (только широкий план):**
→ Нарезать широкий план на перекрывающиеся тайлы того же разрешения что и обучающие кропы (800×600, перекрытие 50%).
→ Прогнать сегментатор по каждому тайлу.

Это предотвращает масштабный сдвиг: модель никогда не видит весь мелкий фасад целиком, только тайлы того же разрешения что при обучении.

## 4.2 Ratios для каждого тайла/кропа

Для каждого тайла или кропа:

```
ratio_X = pixels(class_X) / total_non_ignored_pixels
```

для X из {mechanical, biological, anthropogenic}

## 4.3 Агрегация на уровне здания

Для каждой категории X вычислить по всем тайлам:

- `mean_ratio_X` — средняя плотность повреждений
- `max_ratio_X` — наихудшая локальная концентрация
- `prevalence_X` — доля тайлов где ratio_X > eps

```
prevalence_X = count(тайлы где ratio_X > eps) / total_tiles
```

Рекомендуемое eps примерно 0.005–0.01 (подобрать на валидационной выборке).

**Итоговый вектор признаков (9D):**

```
feature_vector = [
    mean_mechanical,    max_mechanical,    prevalence_mechanical,
    mean_biological,    max_biological,    prevalence_biological,
    mean_anthropogenic, max_anthropogenic, prevalence_anthropogenic
]
```

**Нормализация:** применить `StandardScaler`, обученный на обучающей выборке, перед подачей в модель ранжирования.

⚠ **Известное ограничение**: для тяжёлых зданий (КТС 2–3) сегментационные ratios могут занижать реальный ущерб из-за доменного сдвига (модель обучена преимущественно на умеренных случаях). Калибровка через ранжирование частично компенсирует это. Явно описать в разделе limitations диссертации.

Сохранить в: `building_features.csv`

# Фаза 5 — Датасет попарных сравнений

## 5.1 Интерфейс ранжирования

Простой HTML-инструмент:
- Два **широких плана фасадов** рядом
- Три кнопки: **Левое хуже** / **Правое хуже** / **Примерно одинаково**

Формат вывода: `image_a, image_b, winner` — сохраняется в CSV.

Ранжирование выполняется **только по широким планам фасадов**, никогда по кропам.

## 5.2 Сессии ранжирования

Три независимые сессии (по одной на категорию дефектов):

1. Сравнение механических повреждений
2. Сравнение биологических загрязнений
3. Сравнение антропогенных повреждений

**Целевой объём: 500 пар на категорию** (сокращено с N·log2(N)).

Обоснование: для MLP с 9-мерным входом 500 обучающих пар (около 425 после split 85/15) достаточно для построения стабильной порядковой шкалы. Консервативный старт сохраняет временной бюджет для GNN-стадии. При недостаточном качестве ранкера объём пар можно нарастить.

Итого: примерно **1500 сравнений** (около 4–5 часов работы).

**Ранняя остановка**: после каждой порции из 100 новых пар вычислять Kendall's tau на held-out валидационной выборке. Остановить сбор когда tau улучшается менее чем на 0.01 в двух последовательных порциях.

# Фаза 6 — Обучение модели ранжирования

## 6.1 Разбивка данных

- 85% пар — обучение
- 15% пар — валидация (выделяется с самого начала)

## 6.2 Входное представление

Единое признаковое пространство: **нормализованный 9D вектор** из Фазы 4.

Никаких эмбеддингов. Никаких смешанных представлений. Все здания используют один и тот же тип признаков.

## 6.3 Архитектура

RankNet-образная MLP с разделяемым энкодером и тремя выходными головами.

```
Вход: 9D вектор признаков (для каждого здания в паре)

Разделяемый энкодер (одни веса для обоих зданий в паре):
  Linear(9 -> 32) -> ReLU -> Dropout(0.2)
  Linear(32 -> 16) -> ReLU -> Dropout(0.2)

Латентное представление: 16D

Три независимые выходные головы:
  head_mechanical:    Linear(16 -> 1) -> скалярный score
  head_biological:    Linear(16 -> 1) -> скалярный score
  head_anthropogenic: Linear(16 -> 1) -> скалярный score
```

При обучении: модель получает пару (A, B), предсказывает вероятность того что A хуже B — отдельно для каждой категории.

## 6.4 Функция потерь

```
Loss = BCE(P(A хуже B | mechanical),    label_mechanical)
     + BCE(P(A хуже B | biological),    label_biological)
     + BCE(P(A хуже B | anthropogenic), label_anthropogenic)
```

Опционально: слабая ординальная регуляризация с использованием image-level КТС меток как сигнала согласованности (не как классификационного таргета).

## 6.5 Оценка качества

На валидационных парах:

- Попарная точность (pairwise accuracy)
- ROC-AUC
- Kendall's tau

Дополнительная проверка (sanity check): корреляция Спирмена между предсказанными scores и image-level КТС метками. КТС используются только как внешний ориентир, не как ground truth.

# Фаза 7 — Финальный Damage Index и вход в GNN

## 7.1 Damage Index на основе ранжирования

Для каждого здания прогнать через обученный энкодер RankNet:

```
damage_index = [score_mechanical, score_biological, score_anthropogenic]
```

Каждый score в диапазоне [0, 1], получен из латентного ранжированного пространства.

## 7.2 Гибридный вход в GNN (резервный вариант)

Признак узла GNN объединяет оба представления:

```
node_feature = concat(damage_index, feature_vector)
             = 3D ранговые scores + 9D сегментационные ratios
             = 12D итого
```

**Обоснование**: если модель ранжирования показывает слабое качество (недостаточно пар, слабый сигнал по какой-то категории), сырые сегментационные ratios всё равно дают GNN значимый сигнал. GNN может сама выучить вес каждого представления. Это развязывает качество GNN от качества ранкера.

⚠ Если качество ранкера (Kendall's tau) приемлемо, для входа в GNN может быть достаточно только `damage_index` (3D). 12D вариант — консервативный резервный вариант.

# Ключевые методологические решения (для диссертации)

| Решение | Обоснование |
|---|---|
| Три суперкатегории дефектов | Разные механизмы пространственного распространения в городской среде |
| Единое 9D признаковое пространство | Устраняет проблему смешанных представлений в RankNet |
| Попарное ранжирование как калибратор | Люди надёжнее оценивают относительное состояние чем присваивают абсолютные оценки |
| 500 пар на категорию | Достаточно для 9D MLP; сохраняет временной бюджет для GNN-стадии |
| Гибридный 12D вход в GNN | Развязывает GNN от качества ранкера; резервный вариант надёжности |
| Тайлизация при инференсе | Предотвращает масштабный сдвиг между обучением (кропы) и инференсом (широкие планы) |
| КТС метки только как sanity check | Не смешивает визуальную аппроксимацию со структурным ground truth |

# Конец пайплайна до GNN-стадии (v3)
